## 滑动窗口机制
近保留最近20轮的对话进入prompt，每轮对话检查队列中的条数是否超过20条，如果超过，就丢弃最旧的5条，被丢弃的那5条不再进入prompt。并对“丢弃”的那5条（不从数据库删除）喂给后台摘要模型，通过prompt，让摘要模型结合得到的这5轮对话，以及目前摘要去更新摘要。确保大局观方向不偏离，同时让模型明白目前在哪个任务，在哪个节点，真正的主线是什么
**Prompt 模板：** _“请结合现有的摘要，把这 5 条新对话中的核心进展、未完成的上下文、或者突发的新事件，合并更新进摘要中。请保持字数在 200 字以内。”_
然后小模型输出了一个更新后的摘要，在摘要数据库中新存一个。然后把当前prompt构建要取的摘要替换为最新的这个。

每轮喂给模型的prompt里包含：系统人设+用户问题，以及1.最近的20轮的原始对话 2.这个任务线的最新摘要 3.信号信息 4.用户基础认知信息

# 每一轮对话要给LLM的prompt具体这样设计（都用中文）
### System prompt
- Fixed Layer固定部分（静态）
     - 系统人设（AI助手，性格，它负责的这个软件的功能说明，它所负责的这条任务线的摘要（这由用户为这条线创建智能体的时候写入））
     - 永远不变，应用打开即加载
- context Window 眼前对话区（动态）
     - 最近不超过20轮的原始对话原文。随着用户每进行一轮问答（用户问，LLM答），里面就多一条。超过20条，最老的那条消失。
- 滚动摘要区（动态）
     - 每一轮窗口滑动进行更新，把上一轮的摘要和这次窗口滑动丢弃的那一轮对话给后台模型异步压缩成新一轮的滚动摘要，并同时更新prompt，以及落库
- 动态认知区（动态）
     - 从Mem0搜索最相关的5条标签。根据时间或者相关度之类的，将对用户的认知注入prompt
- 任务信号区（动态）
     - 对于这个任务线的大局观总结，这个任务线是哪条，有哪些支线，现在进行到哪里了，这个可以把数据库里这条任务线（包括所有支线的信息给它），每一轮更新
     - 从Mem0数据库里查询心情信号中最近的3条，对用户当前状态、态度有一个认知。比如如果比较消极就鼓励。比较激动就提醒细节问题

# 数据库存储设计

使用Redis+mem0+SQLite
- Redis用于存储prompt：使用 Redis 的 `Hash` 结构存储，以便快速更新 `system_prompt`，存储当前对话的 `session_state`（但是我不确定Session State有没有跟上面prompt的滚动摘要以及任务信号重合）
- Mem0:Mem0 擅长处理实体的增删改查,当小模型每n（这里我不确定设计为几轮合适）轮异步提取出新的用户偏好（如“用户喜欢极简主义UI”）时，通过 Mem0 的 API 进行 Upsert（更新或插入），保持画像的实时性。不直接给原文，后台先异步调用模型抽取偏好，再由Mem0落库。这里注意要发挥它的用处和功能，不能只是当普通数据库用。
  Mem0的开源版可以手动控制分类逻辑，由我自己的小模型判断每个偏好要放在哪里。再mem0.add的时候作为metadata参数传入
  分类越精确，召回效率就越高
- SQLite (持久化/审计层):
    - **原文与摘要：** 存储完整的 `Raw_Data`（每一轮对话）。这对于后续的 RAG（检索增强生成）召回以及模型微调（Fine-tuning）至关重要。
    - **信号 (Signals)：** 每一轮后台模型生成的摘要
    - 当前任务线：这个很重要，是这个软件的核心功能。可以直接读取这个软件存储的任务线库，确保同步。这个软件的任务线库可能由用户以及智能体体修改。修改的都是软件存储的任务线库。所以这部分我觉得可以分开存储，不跟智能体的SQLite库存在一起
#### 其中还有很关键的一点是信息流转问题
遵循优先落库原则。
用户输入问题，系统在内存用Redis构建prompt/更新prompt，LLM给出回答。给出回答后优先将这一轮的用户问题和LLM回答落库放到智能体的SQLite库，然后后台读取新一轮的内容进行摘要处理、信号处理、用户偏好提取，将这几个分别交给不同的后台模型去做，可以调用便宜一点的API。将这些模型的返回再落库。落库后再读取进内存数据库Redis去更新prompt。然后其中可能小模型返回的内容结构不对什么的，需要采取一些方法。比如强 json schema,或者宽松的正则化机制。
确保信息先落库，再进内存，再去前端显示

#### 还有就是prompt长度约束，为了保证效率，peompt长度约束要控制好。

# Toolcalling
让AI能直接操作我的画布可以自由在里面添加节点。这样我直接告诉他我今天干了啥，它就能帮我写进入
智能体只操作数据库而不直接操作画布，前端代码根据数据变化来自动渲染画布
工程实现上使用Function Calling(工具调用)
为了保证稳定性，采用ReAct模式，进行多轮操作直到确认真实写入。并设置最大轮数、Token消耗数。对每一步都计算Token，但凡Token到达上限或者轮数到达上限就停。
根据LLM输出的Json格式内容去往数据库里增删改查节点。
同时如果我在前端手动增删改查，也要实时同步到后端

然后要求LLM输出json的过程中要用强力的schema，防止各种混乱的格式错误，或者不小心把json输出给用户看。在每轮ReAct中，要把LLM的中间思考过程、自然语言返回都给用户看到。

## 意图识别
对用户给的内容分类，判断是否有修改画布的需求
- 在Router层放个小模型，它的任务是了解我们这个软件（一句话）,并判断用户的话有没有增删改查画布的需求。
- 在SystemPromt里写清楚如何分辨，再做一道防线
- 影子提案机制，不直接修改，而是在AI无法确定时，生成一个影子节点卡片，，调用`propose_node_change` 而不是 `add_node`。根据用户反馈（在节点旁边配两个微型按钮：`√ (确认添加)` 和 `× (忽略)`）去确认要不要写上。
- 能力按需加载。如果是路由为纯聊天，就用上面我设计的聊天prompt(Chat Mode)，如果是要对画布操作，用另一套prompt（Action Mode）.prompt组装需要裁剪掉一些没用的地方，相对于我上面那个模板。可能需要包含系统人设+最近20轮对话+Tool Schema （工具定义）+当前画布状态
- 在操作工具的同时，也要进行有温度的对话：通过 **“任务反馈循环”** 实现，**思维链 (CoT) 引导：** 在 System Prompt 里规定：_“在执行任何 Function 之前，先用 1-2 句话口头同步你的理解和感受。”_，
    - **异步执行流：**
    - AI 回复：“好哒，我明白你的意思了（聊天层），我这就把这个实验步骤加到画布上（意图层）。”
    - **[执行 Function]**
    - AI 再次收到 Function 的结果。
    - AI 最终总结：“搞定啦！你看这个新节点位置合适吗？”
