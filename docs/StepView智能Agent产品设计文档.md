# StepView 智能 Agent 产品设计文档

## 1. 产品一句话

StepView 是一款以任务线、支线、节点日记和成就归档为核心的成长可视化工具，并通过智能 Agent 与分层记忆系统，帮助用户理解自己正在推进的长期任务、分支选择、行动阻塞和真实心理需求。

它不是传统待办清单，也不是泛用聊天助手，而是一个会记住用户任务结构、读懂用户节点日记、陪伴用户持续推进的个人成长地图。

## 2. 背景与需求分析

很多用户并不是没有目标，而是缺少一种能长期承载目标变化的结构。

传统待办工具适合记录“今天要做什么”，但不擅长表达：

- 目标为什么发生变化。
- 中途出现了哪些支线。
- 哪些节点是真正重要的转折。
- 哪些任务虽然没完成，但留下了有价值的经验。
- 用户为什么总是在类似的地方卡住。
- 一段时间后，用户如何理解自己曾经走过的路。

现实中的成长并不是一条笔直的 checklist。它更像一张带有主线、支线、绕路、暂停、回归、放弃和成就的地图。

StepView 的需求洞察是：

- 用户需要的不只是任务管理，而是“把自己走过的路看见”。
- 用户需要的不只是提醒，而是“理解我现在在哪条线、哪个节点、为什么卡住”。
- 用户需要的不只是 AI 回答，而是“基于我的真实任务结构和日记记忆回答”。
- 用户需要的不只是完成感，而是“未完成也能被理解，支线也有意义”。

## 3. 面向人群

### 3.1 学生与自我成长型用户

适合记录学习计划、比赛准备、作品集、实习准备、自学路线和长期能力成长。

这类用户常见痛点：

- 目标很多，但容易混乱。
- 学习过程碎片化，阶段成果不容易被看见。
- 容易因为短期挫败否定长期努力。
- 需要把“我学过什么、卡过什么、突破过什么”沉淀下来。

### 3.2 创作者与个人项目推进者

适合记录内容创作、产品开发、个人品牌、副业验证和长期输出。

这类用户常见痛点：

- 主线和支线很多，容易分散。
- 项目中途经常改变方向。
- 需要复盘哪些尝试值得继续，哪些只是消耗。
- 需要一个能理解分支选择的陪伴型工具。

### 3.3 职业转型与长期规划用户

适合记录职业发展、转行路线、技能补齐、行业研究和阶段性决策。

这类用户常见痛点：

- 不确定性高。
- 决策链条长。
- 需要对多个方向进行低成本验证。
- 需要保留每次选择背后的理由和情绪。

### 3.4 情绪敏感但仍希望向前走的人

StepView 不做医疗意义上的心理诊断，但它重视用户在行动过程中的感受。

这类用户可能需要：

- 不被简单催促。
- 不被粗暴评价效率。
- 允许任务暂停、绕路和支线存在。
- 在节点日记中写下真实情绪，并得到温和理解。

## 4. 产品价值主张

StepView 的核心价值不是“更快完成任务”，而是：

1. 看见路径  
   用户能看见自己从哪里开始，走过哪些节点，产生过哪些支线。

2. 理解分支  
   支线不是干扰，而是现实成长中的探索、关系、合作、犹豫和新机会。

3. 保留过程  
   节点日记让任务不只是状态变化，而是保留当时的想法、情绪和判断。

4. 形成认知  
   Agent 通过摘要和记忆，逐渐理解用户的目标偏好、卡点模式和推进方式。

5. 陪伴推进  
   Agent 的回答绑定任务结构，不脱离用户当前任务线和分支状态。

## 5. 人文关怀点

### 5.1 不把人简化成效率机器

StepView 承认人的成长不是线性的。用户可能暂停、绕路、失败、重启，也可能在某条支线里发现新的意义。

产品不应该只奖励“完成”，也应该尊重：

- 尝试
- 记录
- 犹豫
- 复盘
- 放弃不合适的路线
- 重新选择

### 5.2 支线也值得被看见

很多工具只关心主任务是否完成，但人的真实经历常常发生在支线里。

例如：

- 和朋友一起做过一段尝试。
- 因为喜欢的人开启过一条心动支线。
- 为了验证一个想法临时开了一条探索路线。
- 发现某条分支不值得继续，但它帮助用户做出了判断。

StepView 将支线作为一等结构保存和归档，而不是把它们当成备注。

### 5.3 未完成不等于无意义

未完成的任务线也可能留下重要信息：

- 用户为什么停止。
- 哪个节点产生阻塞。
- 哪个方向被证明成本过高。
- 哪种推进方式不适合用户。

Agent 需要从这些记录里帮助用户理解自己，而不是简单说“你没有坚持”。

### 5.4 节点日记是用户对自己的证词

节点日记不只是文本输入，而是用户在某个时间、某个任务节点上的真实状态记录。

系统应当：

- 保留原文。
- 只做可追溯摘要。
- 不随意上升到人格判断。
- 不做未经授权的深度心理分析。
- 允许用户未来查看、修改和删除记忆。

## 6. 关键产品亮点

### 6.1 任务线可视化

每个目标不再是单个待办项，而是一条可视化任务线。

任务线包含：

- 起点
- 终点
- 里程碑
- 计划节点
- 关键节点
- 完成状态
- 归档记录

### 6.2 支线系统

StepView 支持在任意非终点节点创建支线。

支线类型包括：

- 我的分支
- 伙伴支线
- 恋人支线

支线可以延展，也可以接回主线。任务完成后，支线会随任务一起进入归档，而不是留在活跃画布上造成数据不一致。

### 6.3 节点日记

用户可以在节点里记录：

- 当时发生了什么。
- 为什么做这个选择。
- 卡在哪里。
- 情绪如何。
- 下一步想怎么做。

Agent 会从这些内容中提取轻量信号，例如情绪、卡点、主题和目标变化。

### 6.4 成就与归档

完成任务后，用户可以在 Wins / Journey 中回看完整路径。

归档内容包括：

- 主线节点
- 里程碑
- 计划节点
- 支线记录
- 支线节点
- 完成时间
- 相关统计

归档不是简单把任务移出画布，而是生成可回看的成长档案。

### 6.5 智能 Agent

Agent 能读取当前任务结构，并支持按范围提问：

- 全局
- 某条任务线
- 某条支线

用户可以问：

- “我最近整体状态怎么样？”
- “副业这条线接下来怎么走？”
- “内容号这个分支还值得继续吗？”
- “我是不是总在类似地方卡住？”

Agent 不直接扫描全量历史，而是优先读取轻量摘要和状态快照。

### 6.6 本地与 OpenAI 双模式

StepView 当前支持：

- 本地 Agent 模式
- OpenAI 增强模式

本地模式轻量、快速、离线可用。OpenAI 模式由用户主动开启，并通过设置页填写 API Key。

默认策略是：

- 日常陪伴走本地。
- 深度分析按需使用 OpenAI。
- 不把 API Key 写入 board 数据。
- 不把完整画布直接发送给模型。

## 7. Agent 记忆系统设计

### 7.1 设计原则

Agent 记忆不是简单保存聊天记录，而是围绕任务结构形成可检索、可压缩、可追溯的认知系统。

原则：

- 原文保留。
- 摘要派生。
- 局部更新。
- 范围检索。
- 证据可追溯。
- 深度分析按需触发。

### 7.2 记忆分层

#### 原始层

保存用户真实数据：

- 任务线
- 节点
- 支线
- 节点详情
- 节点日记
- 成就
- 归档支线

#### 信号层

从节点内容中提取轻量信号：

- 情绪倾向
- 主题
- 卡点
- 决策变化
- 目标变化

#### 摘要层

生成用于低消耗问答的摘要：

- 任务线摘要
- 支线摘要
- 最近节点记录
- 当前状态快照

#### 认知层

形成长期用户认知：

- 推进偏好
- 常见卡点
- 决策风格
- 压力来源
- 任务选择倾向

认知层必须带置信度和证据来源，不能凭单次记录下结论。

### 7.3 当前实现

当前代码中 Agent 记忆由 `src/agentMemory.js` 负责。

已实现的数据结构包括：

- `taskLineSummaries`
- `branchMemories`
- `diarySignals`
- `userMemoryFacts`
- `userStateSnapshot`

当前问答流程：

```text
用户问题
  -> 选择范围
  -> 构建 agentMemory
  -> 轻量路由
  -> 本地回答
  -> 可选 OpenAI 增强回答
```

### 7.4 CPU 轻量化策略

StepView 默认不依赖重推理。

轻量化策略：

- 不全量扫描长历史。
- 不把完整 board 发给模型。
- 优先读取摘要。
- 支线和任务线按需检索。
- 最近日记信号限制数量。
- OpenAI 增强由用户主动触发。
- API Key 由用户在设置页填写。

当前发送给 OpenAI 的内容只包括：

- 用户问题
- 路由范围
- 本地回答草稿
- 用户状态快照
- 任务线摘要
- 支线记忆
- 最近日记信号
- 用户认知事实

## 8. 设置页面与 API Key

StepView 提供设置页面，让用户自行配置 OpenAI API Key。

设置项包括：

- 默认 Agent 模式：本地 / OpenAI
- OpenAI API Key
- 默认模型

安全边界：

- API Key 不写入 `stepview-board.json`。
- API Key 当前保存在本机 `localStorage`。
- 后续可升级为 Electron 主进程加密存储或系统凭据管理。

## 9. 工程实现细节

### 9.1 前端技术

当前使用：

- React
- Vite
- CSS
- Electron
- Vitest

### 9.2 核心文件

- `src/main.jsx`
  - 主界面。
  - 任务画布。
  - 支线交互。
  - Agent 面板。
  - 设置页面。

- `src/progressCore.js`
  - 任务、节点、支线、归档、成就等核心数据逻辑。

- `src/agentMemory.js`
  - Agent 记忆构建。
  - 轻量路由。
  - 本地回答生成。

- `electron/main.js`
  - Electron 主进程。
  - 本地文件存储。
  - OpenAI API 请求代理。

- `electron/preload.js`
  - 渲染进程与主进程桥接。

- `electron/boardStorage.js`
  - board 文件读写。
  - 备份文件。
  - 原子写入。

### 9.3 数据持久化

桌面端数据保存到 Electron 用户数据目录中的 `stepview-board.json`。

存储内容包括：

- tasks
- stickers
- links
- branches
- achievements
- agentMemory
- updatedAt

API Key 不保存在该文件中。

### 9.4 支线归档

完成任务时：

1. 任务状态变为 completed。
2. 与该任务相关的活跃支线从 `board.branches` 移除。
3. 支线快照进入任务自身的 `archivedBranches`。
4. 归档页读取 `archivedBranches` 展示支线历史。
5. 恢复任务时，支线重新回到活跃 `branches`。

这样避免了“任务完成但支线仍在画布上”的不一致。

### 9.5 OpenAI 增强链路

OpenAI 调用发生在 Electron 主进程，避免直接在前端暴露请求逻辑。

流程：

```text
前端 Agent 面板
  -> 生成本地回答和 agentMemory
  -> ipcRenderer 调用 askOpenAI
  -> Electron 主进程请求 OpenAI Responses API
  -> 返回增强回答
  -> 前端展示
```

失败时：

- 保留本地回答。
- 显示失败原因。
- 不阻断用户继续使用。

## 10. STAR 法则表达

STAR 法则适合用来讲 StepView 的产品价值、项目经历和功能设计。

### 10.1 Situation 背景

用户在长期成长和复杂任务推进中，经常面对多目标、多分支、反复暂停和情绪波动。传统待办工具更关注任务是否完成，却难以承载任务背后的过程、选择和个人状态。

### 10.2 Task 任务

StepView 要解决的问题是：如何让用户不仅能管理目标，还能看见自己的成长路径，理解支线探索的价值，并通过智能 Agent 获得基于真实任务结构的陪伴和建议。

### 10.3 Action 行动

产品采取了以下设计：

- 用可视化画布承载任务线。
- 用节点记录里程碑、计划和关键时刻。
- 用支线表达探索、合作和关系中的分支路径。
- 用节点日记保存用户当时的真实想法。
- 用归档系统完整保留主线和支线。
- 用 Agent 记忆系统提取任务线摘要、支线记忆、日记信号和用户认知。
- 用本地模式保证低消耗和离线可用。
- 用 OpenAI 增强模式支持更深层的分析和表达。

### 10.4 Result 结果

StepView 最终形成了一个区别于普通待办工具的体验：

- 用户能看见自己的完整成长路径。
- 支线不再丢失，也能被归档和复盘。
- 节点日记可以被转化为可用记忆。
- Agent 能按任务线和支线回答，而不是泛泛聊天。
- 产品在 CPU 环境下仍能保持轻量运行。
- AI 能力成为增强层，而不是压垮主流程的负担。

## 11. 产品差异化

与普通待办工具相比：

- StepView 强调路径，而不是列表。
- StepView 强调支线，而不是单线程任务。
- StepView 强调日记和认知，而不是单纯状态。
- StepView 强调陪伴，而不是催促。

与普通 AI 聊天工具相比：

- StepView 的 Agent 有任务结构。
- StepView 的 Agent 有支线记忆。
- StepView 的 Agent 有节点日记来源。
- StepView 的 Agent 可以按范围回答。
- StepView 的 Agent 默认轻量，不依赖全量上下文。

## 12. 后续路线

### 12.1 短期

- 将节点详情拆分为节点说明和节点日记。
- 增加用户可查看、编辑、删除的记忆管理界面。
- 给 Agent 增加更明确的“普通陪伴 / 深度分析 / 行业调研”模式。
- 增强任务线和支线的自动识别能力。

### 12.2 中期

- 支持周期性成长报告。
- 支持跨任务线模式识别。
- 支持更多 AI Provider。
- 支持 OpenAI 流式输出。
- 支持更安全的 API Key 存储方式。

### 12.3 长期

- 形成真正的个人成长知识库。
- 将用户的任务线、分支选择、日记、成就和长期认知连接起来。
- 帮助用户从“我做了什么”走向“我正在成为什么样的人”。

## 13. 总结

StepView 的核心不是管理任务，而是让用户看见自己。

它把任务线变成地图，把支线变成探索，把日记变成记忆，把完成变成归档，把 AI 变成一种不脱离上下文的陪伴。

当用户问“我现在该怎么走”时，StepView 不应该给出空泛建议，而应该能看见：

- 用户在哪条任务线上。
- 用户卡在哪个节点。
- 哪条支线正在影响判断。
- 最近日记里反复出现了什么。
- 过往成就说明用户其实已经走了多远。

这就是 StepView 智能 Agent 的产品方向：不是替用户生活，而是帮助用户理解自己的路径，并温和地继续向前。
